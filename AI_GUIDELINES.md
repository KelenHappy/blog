## 項目概述

- **項目類型**: Hugo 靜態網站生成器 Blog
- **語言**: 繁體中文
- **主要目錄**:
  - `content/posts/` - 文章存放位置
  - `config/` - 站點配置
  - `themes/` - Hugo 主題
  - `assets/` - 資源文件

## AI 責任範圍

### ✅ AI 應該做的

- 撰寫新文章（高質量、邏輯清晰）
- 編輯和改進現有文章
- 創建或修改配置文件
- 修復 Markdown 格式問題
- 優化文章 SEO
- 測試文章（使用 `hugo server`）
- 管理文件結構
- 提供 Git 提交消息建議

### ❌ AI 不應該做的

- **進行任何 Git 操作**（git add, commit, push, pull 等）
- 刪除用戶手動創建的內容（除非明確指示）
- 修改 Git 歷史記錄
- 直接修改主題代碼（除非是 bug fix）
- 改變項目的基本架構

## 文章撰寫標準

### 命名規則

```
YYYY-MM-DD-kebab-case-title.md
例: 2024-06-26-how-to-use-hugo.md
```

### 必須包含的 Front Matter

```yaml
---
title: "文章標題"
date: 2024-06-26
categories: [分類]
tags: [標籤1, 標籤2]
draft: false
description: "50-160 字的 SEO 描述"
---
```

### 文章結構要求

1. **長度**: 最少 300 字，推薦 800-2000 字
2. **結構**:
   - 開篇引論（50-100 字）
   - 2-4 個主要段落（用 `## 標題`）
   - 每個段落 3-5 句話
   - 代碼示例（如適用）
   - 總結段落

3. **格式要求**:
   - 標題層級: `##` (一級), `###` (二級)
   - 列表項用 `-` 和 2 個空格縮進
   - 代碼塊必須指定語言
   - 段落間用空行分隔

### 寫作風格

- 使用繁體中文（台灣用語）
- 清晰、直接、易於理解
- 避免冗長句子（最多 40 字/句）
- 使用 Markdown 語法增強可讀性
- 適當使用 emoji 增加視覺效果

## 技術操作規範

### 本地測試

撰寫或修改文章後，**必須**在本地測試：

```bash
cd /home/kelen/文件/GitHub/blog
hugo server
# 訪問 http://localhost:1313 查看效果
```

確認以下事項：
- [ ] 文章正常顯示
- [ ] 格式沒有問題
- [ ] 圖片正確加載（如有）
- [ ] 連結有效
- [ ] 代碼高亮正常

### 圖片處理

- **保存位置**: `content/posts/images/`
- **命名**: `YYYY-MM-DD-descriptive-name.jpg`
- **優化**:
  - JPG: 質量 80-85，大小 < 500KB
  - PNG: 使用時進行壓縮
  - 推薦工具: ImageOptim, TinyPNG
- **引用**: 使用相對路徑 `![alt](images/filename.jpg)`

### 文件操作

- **创建**: 使用 `create` 工具
- **編輯**: 使用 `edit` 工具進行外科手術式修改
- **刪除**: 確認後使用 `bash` 工具

```bash
# 示例：創建新文章
touch /home/kelen/文件/GitHub/blog/content/posts/2024-06-26-new-post.md
```

## Git 工作流

### AI 的 Git 責任

**AI 完全不進行任何 Git 操作。** 

所有 Git 操作由用戶負責：
- `git add`
- `git commit`
- `git push`
- `git pull`
- 任何其他 Git 命令

### AI 的角色

當完成工作時，AI 會：
1. 列出修改的檔案
2. 建議合適的提交消息（供參考）
3. 提醒用戶需要進行 Git 提交

## 常見任務操作指南

### 任務 1: 撰寫新文章

```
步驟 1: 確定文章主題和日期
步驟 2: 創建檔案 content/posts/YYYY-MM-DD-title.md
步驟 3: 撰寫高質量內容
步驟 4: 本地測試 (hugo server)
步驟 5: 通知用戶，建議的提交消息為 "feat(posts): ..."
```

### 任務 2: 編輯現有文章

```
步驟 1: 使用 edit 工具進行修改
步驟 2: 更新 lastmod 字段為當前日期
步驟 3: 本地測試驗證
步驟 4: 通知用戶，建議的提交消息為 "fix(posts): ..."
```

### 任務 3: 添加圖片到文章

```
步驟 1: 優化圖片（< 500KB）
步驟 2: 保存到 content/posts/images/
步驟 3: 在文章中添加 ![alt](images/filename.jpg)
步驟 4: 本地測試
步驟 5: 通知用戶完成，建議的提交消息
```

### 任務 4: 更新配置

```
步驟 1: 定位正確的配置文件
步驟 2: 使用 edit 工具進行修改
步驟 3: 測試新配置
步驟 4: 通知用戶完成，建議的提交消息為 "fix(config): ..."
```

## AI 工作流建議

### AI 工作流建議

```
1. 理解用戶需求 → 確認明確
2. 規劃工作 → 列出步驟
3. 執行任務 → 使用工具（create/edit）
4. 驗證結果 → 本地測試
5. 通知用戶 → 說明完成情況和建議的 Git 消息
```

### 使用工具的優先順序

1. **查看文件**: `view` 工具
2. **編輯文件**: `edit` 工具（已存在）或 `create`（新建）
3. **查詢**: `grep`, `glob` 工具
4. **複雜操作**: `bash` 工具

### 錯誤處理

如果遇到問題：
1. 檢查錯誤消息
2. 查看相關文件
3. 嘗試修復（或回滾）
4. 向用戶解釋並提供解決方案

- [ ] 內容質量高（無語法/拼寫錯誤）

- [ ] 已本地測試（hugo server）
- [ ] Git 提交消息規範
- [ ] 沒有臨時或調試代碼
- [ ] 檔案已正確保存

## 限制和邊界

### 不應修改

- Git 歷史記錄（rebase, force push）
- 主題核心代碼（除非 bug fix）
- 用戶的個人喜好設置
- CI/CD 配置（未經指示）

### 需要用戶確認的操作

- 刪除文章或檔案
- 修改重要配置
- 進行大規模重構
- 更改項目結構
